# VidScreener Database Schema

The VidScreener platform utilizes Supabase (PostgreSQL) as its primary relational store. Below is an overview of the core schema configuration and relationships.

## Core Tables

### `organizations`
Stores tenant information (useful if scaling to B2B SaaS in the future).
- `id` (uuid, pk)
- `name` (text)
- `created_at` (timestamptz)

### `users` (Managed by Supabase Auth)
Stores core authentication identities. We use a secondary profile table to map users to roles within the application.

### `evaluators` (Profiles)
Extends the Auth User with application-specific metadata and role assignment.
- `id` (uuid, pk, references auth.users)
- `organization_id` (uuid, fk)
- `name` (text)
- `email` (text)
- `role` (enum: 'admin', 'evaluator')
- `created_at` (timestamptz)

### `projects`
Represents an overarching evaluation campaign (e.g., "Summer Internships 2026").
- `id` (uuid, pk)
- `organization_id` (uuid, fk)
- `name` (text)
- `description` (text)
- `status` (enum: 'draft', 'active', 'archived')
- `rubric_json` (jsonb) - The schema the AI evaluates against.
- `created_at` (timestamptz)

### `forms`
Configuration for the unauthenticated applicant pipeline.
- `id` (uuid, pk)
- `project_id` (uuid, fk)
- `title` (text)
- `prompt_text` (text)
- `max_duration_seconds` (int)
- `is_active` (boolean)
- `created_at` (timestamptz)

### `submissions`
A record of an applicant submitting metadata to a specific form.
- `id` (uuid, pk)
- `form_id` (uuid, fk)
- `applicant_name` (text)
- `applicant_email` (text)
- `status` (enum: 'pending_video', 'processing', 'under_review', 'completed')
- `created_at` (timestamptz)

### `videos`
Metadata for the securely stored video in MinIO.
- `id` (uuid, pk)
- `submission_id` (uuid, fk)
- `project_id` (uuid, fk)
- `storage_path` (text) - The MinIO bucket object key.
- `original_filename` (text)
- `duration_seconds` (int)
- `created_at` (timestamptz)

### `evaluator_assignments`
Maps a human evaluator to a specific video for review.
- `id` (uuid, pk)
- `video_id` (uuid, fk)
- `evaluator_id` (uuid, fk)
- `status` (enum: 'pending', 'in_progress', 'completed')
- `due_date` (timestamptz)
- `created_at` (timestamptz)

### `evaluations`
Stores actual review data. Can be generated by AI or Humans.
- `id` (uuid, pk)
- `video_id` (uuid, fk)
- `evaluator_id` (uuid, fk, NULL if AI)
- `assignment_id` (uuid, fk)
- `evaluation_type` (enum: 'ai_initial', 'human_override', 'human_approval')
- `scores_json` (jsonb) - The array of scores mapping back to the Rubric.
- `overall_score` (int)
- `flags_json` (jsonb) - Any critical issues.
- `notes` (text)
- `created_at` (timestamptz)

## Security & Row Level Security (RLS)
The database enforces tenant isolation and domain boundaries using Supabase RLS.

- `read`: Evaluators can only query `evaluator_assignments` and `videos` where `evaluator_assignments.evaluator_id = auth.uid()`.
- `insert`: The frontend Applicant Portal uses a locked-down Anon key capable only of inserting into `submissions` and `videos` with restricted payload footprints.
- `admin`: Users with the role 'admin' in the `evaluators` table possess bypass capabilities to fetch aggregate scopes for the admin dashboard APIs.
